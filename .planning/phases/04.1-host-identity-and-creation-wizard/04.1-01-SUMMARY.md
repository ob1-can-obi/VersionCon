# Phase 4.1 Plan 01: Types + Protocol for Host Pre-Registration Summary

Adds the `HostIdentity` type (memberId/displayName/hostAuthSecret) and an optional `AuthRequest.hostAuthSecret?: string` wire field — the structural surface plans 04.1-02 (SessionHost host pre-registration) and 04.1-03 (WizardPanel + loopback wiring) will build on. Purely additive: no behavior changes, no SessionHost edits, no WizardPanel edits.

## Objective Recap

Phase 4.1 closes two architectural defects discovered during Phase 4 multi-window UAT:

- **Defect A:** Wizard never collects `displayName` (closed in plan 04.1-03).
- **Defect B:** First-authenticated WebSocket wins host role (closed in plan 04.1-02 by pre-allocating a `HostIdentity` in the wizard, threading it into `new SessionHost(config, hostIdentity)`, and gating role:'host' on `claimedMemberId === hostMemberId && claimedHostAuthSecret === hostAuthSecret`).

Plan 04.1-02 cannot land without `HostIdentity` and `AuthRequest.hostAuthSecret?` — those are this plan's deliverables. Wave-1 lands them in a single commit so 04.1-02 and 04.1-03 run in parallel against a stable surface.

### Task 1: HostIdentity type

Appended `export interface HostIdentity { memberId: string; displayName: string; hostAuthSecret: string; }` after `SessionConfig` in `src/types/session.ts`. JSDoc documents the security boundary: `hostAuthSecret` is the sole gate for `role:'host'` (paired with memberId match) and is NEVER serialized to the wire except in the host's own loopback auth-request frame. Purely additive — `SessionConfig`, `Member`, `Session`, `SavedSession`, `TransportAdapter`, `ConnectionStatus`, `SessionRole` all preserved verbatim.

### Task 2: AuthRequest.hostAuthSecret?

Added optional field `hostAuthSecret?: string` to the existing `AuthRequest` interface in `src/network/protocol.ts`. Two JSDoc blocks (one on the interface, one on the field) document:

- The field is populated ONLY by the host's loopback SessionClient (plan 04.1-03 wires it).
- Server-side validation that gates `role:'host'` lands in plan 04.1-02 (`claimedMemberId === host.hostMemberId && hostAuthSecret === host.hostAuthSecret`).
- Backwards-compat: undefined field = JSON.stringify omits the property entirely.

`MessageType` discriminator, `ProtocolMessage` union, `parseMessage`, and `sendMessage` all unchanged.

### Task 3: Round-trip tests

Appended a new suite `'AuthRequest hostAuthSecret round-trip (Phase 4.1)'` to the existing `src/test/suite/protocol.test.ts` (and added `type AuthRequest` to the imports already pulled from `../../network/protocol.js`). Three tests:

1. **Backwards-compat:** AuthRequest WITHOUT `hostAuthSecret` round-trips through `JSON.stringify` → `parseMessage`; parsed message has `hostAuthSecret === undefined`.
2. **Forward-compat:** AuthRequest WITH `hostAuthSecret = 'secret-uuid-abc-123'` round-trips and the parsed message preserves the secret verbatim.
3. **Tolerant parse on malformed type:** A frame with `hostAuthSecret: { evil: 'object' }` parses without throwing; type-discriminator preserved. Server-side `typeof === 'string'` check is plan 04.1-02's responsibility.

## Known Stubs

None — this is type/protocol prep, not behavior. `AuthRequest.hostAuthSecret` is a wire field with no server-side reader yet (plan 04.1-02 adds the reader). That is the documented intended state, not a stub: a remote client setting `hostAuthSecret` in their own auth-request without server-side validation is a no-op (no field on the host reads it yet), which is the correct posture for a Wave-1 type-prep plan.

## Decisions Made

1. **Optional `hostAuthSecret?: string` over a separate `AuthHostRequest` interface** — keeps the wire-type discriminator stable (`type: 'auth-request'`), allows backwards-compat with pre-Phase-4.1 clients without bumping a protocol version. Trade: type-narrowing burden on plan 04.1-02's `handleAuthRequest` (must `typeof === 'string'` guard).
2. **`HostIdentity` placed in `src/types/session.ts` (not a new file)** — follows the existing convention that all session-shape types live in one place (`Member`, `SessionConfig`, `Session`, `SavedSession`). One module, one mental model.
3. **Three required string fields on `HostIdentity` — no optional fields.** WizardPanel allocates all three before constructing SessionHost; partial allocations are a programmer error and should fail at the type-checker, not at runtime.

## Downstream Impact

- **Plan 04.1-02 (SessionHost host pre-registration)** can now `import type { HostIdentity } from '../types/session.js'` and accept it in `new SessionHost(config, hostIdentity)`. Server-side `handleAuthRequest` can read `msg.hostAuthSecret` from the parsed `AuthRequest` to gate `role:'host'`.
- **Plan 04.1-03 (WizardPanel + loopback wiring)** can construct a `HostIdentity` (UUIDs from `crypto.randomUUID()` + user-confirmed displayName from wizard step 1) and pass `hostAuthSecret` through the loopback `SessionClient`'s auth-request.
- Both downstream plans run in parallel against the surface this plan establishes.
