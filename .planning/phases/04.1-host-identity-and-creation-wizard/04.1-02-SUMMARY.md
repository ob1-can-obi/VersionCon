## Accomplishments

- **Defect B closed.** The "first-authenticated WebSocket wins host role" race in `SessionHost.ts:501-505` is gone. handleAuthRequest now checks `crypto.timingSafeEqual(claimedSecret, this.hostAuthSecret)` with a length pre-check; remote clients without the secret (or with a guess) always receive `role:'member'` regardless of timing. The host's loopback SessionClient (plan 04.1-03's responsibility) is the only path to `role:'host'`.
- **SessionHost constructor signature widened.** `constructor(config, hostDisplayName: string)` → `constructor(config, hostIdentity: HostIdentity)`. The wizard / extension.ts loopback wiring (plan 04.1-03) generates the triple via `crypto.randomUUID()` for memberId and hostAuthSecret + the user-confirmed displayName from wizard step 1.
- **Race window structurally closed.** `this.hostMemberId` is set in the constructor body from `hostIdentity.memberId` BEFORE start() binds the WebSocket listener. Synchronous construction + start() ordering means no remote client can ever observe `hostMemberId === null` (the old branch's discriminator). The pre-allocated UUID is overwritten at the moment of host auth with the loopback's ws-authed memberId so admin gates (`requesterId !== this.hostMemberId`) resolve against the live connection.
- **`hostMemberId` field nullability preserved.** The field type stays `string | null` to preserve the existing 04-04 removeMember reset on host disconnect (admin commands fail-close after host disconnect).
- **All Phase 4 closures preserved.** CR-01-NEW (broadcasts use `record.memberDisplayName`, not `this.hostDisplayName`), CR-02-NEW (extension.ts host-self echo at 6+ sites untouched), CR-03-NEW (segment-aware path-traversal validator unchanged), 04-13 CR-01 (chat-message `kind: 'user'` coercion unchanged) — all verified by greps in the validation cascade.
- **Six regression tests added.** New `Phase 4.1 host pre-registration` suite covers: (1) loopback host with correct secret → `role:'host'`, (2) remote without secret → `role:'member'`, (3) remote with wrong-length secret → `role:'member'` + no throw, (4) first-to-connect remote → `role:'member'` (race window closed), (5) loopback host kickMember → remote receives `member-kicked` (admin gate continuity), (6) `broadcastPush` with remote PushRecord still uses `record.memberDisplayName`, NOT `HOST_NAME` (CR-01-NEW preservation regression guard).

## Decisions Made

1. **Pre-allocate in constructor (not in start()).** Synchronous construction runs strictly before start(), eliminating any race between the listener bind and the first inbound auth. This is the structural reason Defect B closes — the race-window discriminator (`hostMemberId === null`) is unreachable from any external trigger.
2. **Re-point `this.hostMemberId` at successful host auth.** Preserves admin-gate semantics (`requesterId !== this.hostMemberId`) without auditing every gate. The pre-allocated UUID is a placeholder until first host auth.
3. **Keep `this.hostMemberId` nullable (`string | null`).** Preserves the existing 04-04 `removeMember` reset on host disconnect (admin commands fail-close after host disconnect). Multi-host / host migration is explicitly out of scope per ROADMAP Phase 4.1 (deferred to backlog 999.4).
4. **Length pre-check before `crypto.timingSafeEqual`.** Avoids the throw on length-mismatched buffers. The length leak is bounded to `crypto.randomUUID()`'s publicly-known 36-char form; no additional surface.
5. **Defense-in-depth: `typeof === 'string'` guard before `Buffer.from`.** parseMessage is permissive (returns whatever JSON.parse produces; pinned by plan 04.1-01 Test 3); the typeof guard at the consumer is the matching enforcement layer. A non-string `msg.hostAuthSecret` never reaches `Buffer.from` or `timingSafeEqual`.

## Threat Model Compliance

The plan's STRIDE register has 5 entries (T-04.1-02-01 through T-04.1-02-05). All 3 mitigate-disposition entries are addressed in code:

- **T-04.1-02-01 (Spoofing — remote claims role:'host'):** `crypto.timingSafeEqual` after length pre-check; 122-bit secret + 5/min/IP rate limit make brute-force statistically intractable. Test C in the new suite asserts a wrong-length attempt is rejected without throw.
- **T-04.1-02-02 (Tampering — non-string hostAuthSecret):** `typeof claimedSecret === 'string'` guard before `Buffer.from`. parseMessage's permissive contract (pinned by 04.1-01 Test 3) is matched by this consumer-side enforcement.
- **T-04.1-02-03 (Information disclosure — secret leaked in logs/errors):** No log line on rejection paths (typeof / length / timingSafeEqual all silently fall through to `role = 'member'`). Auth-response is unchanged (`accepted: true` because invite-code is the auth gate, not the secret), so an attacker has no oracle distinguishing right-secret from wrong-secret except via the assigned role in subsequent messages — intended.

The 2 accept-disposition entries (T-04.1-02-04 repudiation after host disconnect, T-04.1-02-05 DoS via flood) are scope-deferred per the plan.
