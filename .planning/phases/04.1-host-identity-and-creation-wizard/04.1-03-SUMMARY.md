# Phase 4.1 Plan 03: WizardPanel + extension.ts displayName + HostIdentity Plumbing Summary

Closes Defect A (wizard never collected displayName — host appeared as the literal "Host" without UI affordance) and completes the WizardPanel-side plumbing for Defect B (host identity pre-allocated by the wizard, threaded through to SessionHost.constructor and to extension.ts at module scope). Server-side gating of `role:'host'` by `claimedMemberId === hostMemberId && claimedHostAuthSecret === hostAuthSecret` is plan 04.1-02's responsibility (running parallel in Wave 2).

## Objective Recap

Phase 4.1 closes two architectural defects discovered during Phase 4 multi-window UAT:

- **Defect A:** Wizard never collects `displayName` → closed by this plan. Step 1 widens to collect `{ sessionName, displayName }`. Default chain: workspace settings → `git config user.name` → `os.userInfo().username` → 'Host'. Validation: trim, non-empty, ≤64 chars, no control chars. Workspace-scoped persistence on advance.
- **Defect B (plumbing only here):** First-authenticated WebSocket wins host role. This plan pre-allocates `HostIdentity = { memberId: crypto.randomUUID(), displayName, hostAuthSecret: crypto.randomUUID() }` in `WizardPanel.handleWizardComplete` and passes it to `new SessionHost(config, hostIdentity)`. The server-side validation gate that rejects mismatched claims lands in plan 04.1-02 (parallel Wave 2).

Wave 2 parallel plan handoff: this plan touches `src/ui/WizardPanel.ts`, `src/ui/webview/wizard/wizard.js`, `package.json`, `src/extension.ts`. Plan 04.1-02 touches `src/host/SessionHost.ts`, `src/test/suite/host.test.ts`. The two plans together produce a tsc-clean tree; in isolation each plan errors at the call site the other plan owns. The orchestrator runs full tsc/test after both worktrees merge.

### Task 2: WizardPanel.ts — displayName field, default chain, validation, persistence, HostIdentity allocation

Five coordinated changes inside `src/ui/WizardPanel.ts`:

1. **Imports added.** `import * as os from 'os';` and `import * as childProcess from 'child_process';`. The existing `import type { SessionConfig } from '../types/session.js';` widened to `import type { HostIdentity, SessionConfig } from '../types/session.js';`.

2. **WizardState extended.** New field `displayName: string;` after `sessionName` (both are upfront-identity concerns).

3. **Default-resolution helper added.** New private static method `WizardPanel.resolveDefaultDisplayName(): string` implementing the four-step chain:
   - **Step 1:** `vscode.workspace.getConfiguration('versioncon').get<string>('displayName')` if non-empty after trim
   - **Step 2:** `childProcess.execFileSync('git', ['config', 'user.name'], { cwd: workspaceFolder, timeout: 1000, encoding: 'utf-8', stdio: ['ignore', 'pipe', 'ignore'] })` wrapped in try/catch
   - **Step 3:** `os.userInfo().username` wrapped in try/catch (defensive against sandboxes)
   - **Step 4:** literal `'Host'`

   Defensive: any thrown error in step 2 (ENOENT, timeout, non-git workspace, non-zero exit) silently falls through to step 3. The cwd is set to the first workspace folder's `uri.fsPath` (or `process.cwd()` if no workspace) so a machine-wide git config is honored. `stdio` detaches stdin/stderr; only stdout is captured.

4. **handleWizardNext case 1 widened.** Validates BOTH sessionName (existing 100-char cap, non-empty trim) AND displayName (new). displayName validation:
   - trim, non-empty → `'Display name is required.'`
   - length > 64 → `'Display name must be 64 characters or fewer.'`
   - regex `/[ -]/` matches → `'Display name cannot contain control characters.'`

   On all three error paths, `state.error` is set and the wizard re-renders without advancing the step (mirrors existing sessionName validation pattern). On success, both `state.sessionName` and `state.displayName` are updated, AND the displayName is persisted to workspace settings via `vscode.workspace.getConfiguration('versioncon').update('displayName', displayName, vscode.ConfigurationTarget.Workspace)`. Persistence failures are non-fatal (`.then(undefined, () => {})` swallows promise rejections — the in-memory `state.displayName` is the source of truth for the active session; a failed settings update only means the next session in this workspace won't pre-fill).

   The control-char regex was specifically authored in the unambiguous escape form `/[ -]/` (NOT the alternate `/[<literal NUL>-<literal US><literal DEL>]/` form). Acceptance grep enforced this: the literal-byte form was caught and replaced via a Python heredoc (binary-safe replacement).

5. **handleWizardComplete pre-allocates HostIdentity.** The previous `vscode.workspace.getConfiguration('versioncon').get<string>('displayName') ?? 'Host'` read at WizardPanel.ts:426-429 was REMOVED. New allocation:

   ```typescript
   const hostIdentity: HostIdentity = {
     memberId: crypto.randomUUID(),
     displayName: this.state.displayName,
     hostAuthSecret: crypto.randomUUID(),
   };
   this.sessionHost = new SessionHost(config, hostIdentity);
   ```

   Two random UUIDs (122 bits each — UUIDv4 from Node's `crypto.randomUUID()`). The triple is passed to `new SessionHost(config, hostIdentity)` — matching plan 04.1-02's widened constructor signature. The `onSessionStartedCallback` is then invoked with three args: `(this.sessionHost, this.state.sessionName, hostIdentity)`.

   The `hostAuthSecret` is allocated once, passed once to `SessionHost` and once to the callback. It is NEVER logged (verified by grep: 0 console.log/error/warn/info hits, 0 vscode.window.show*Message hits) and never displayed in toasts. The wizard panel does not retain the secret beyond the session lifetime — `dispose()` is unchanged but the panel's `state` no longer holds the secret (only the locally-scoped `hostIdentity` const inside `handleWizardComplete` does, and it's GC-eligible after the method returns).

6. **Callback type fields widened.** Three signature updates:
   - `onSessionStartedCallback` private field type: `((host, sessionName, hostIdentity: HostIdentity) => void) | null`
   - `createOrShow` static method's `onSessionStarted?` parameter
   - private `constructor`'s `onSessionStarted` parameter

5. **versioncon.hostSession callback widened.** The `WizardPanel.createOrShow(context, callback)` invocation now provides a callback with three args: `(host, sessionName, hostIdentity) => wireHostEvents(host, sessionName, hostIdentity)`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Control-char regex literal-byte → escape form correction**
- **Found during:** Task 2 (handleWizardNext case 1 validation).
- **Issue:** When the Edit tool wrote the regex `/[ -]/` from my heredoc, the `\uXXXX` escape sequences inside the string literal were NOT preserved as escape sequences in the destination file — instead, they were interpreted as literal control characters (NUL `0x00`, US `0x1f`, DEL `0x7f`) and written as raw bytes. Hex-dump of line 424 confirmed: `2f5b 002d 1f7f 5d2f` (`/[<NUL>-<US><DEL>]/`). The plan's acceptance grep `grep -c "\\\\u0000-\\\\u001F" src/ui/WizardPanel.ts returns 1` was specifically authored to reject this broken form.
- **Fix:** Used a Python heredoc with binary-safe `bytes.replace()` to swap the literal-control-char regex (`/[\x00-\x1f\x7f]/`) for the unambiguous escape form (`/[\\u0000-\\u001F\\u007F]/`). This is the form intended by the plan: TypeScript regex literals interpret `\uXXXX` as a unicode escape, so the runtime semantics are equivalent, but the source-code form is auditor-friendly (no invisible control bytes). Verified post-fix via `sed -n '424p'`: `if (/[ -]/.test(displayName)) {`.
- **Files modified:** src/ui/WizardPanel.ts (line 424 only)
- **Commit:** Folded into commit 313d362 (Task 2 — pre-commit fix)

No deviations from plan logic; the deviation was a tooling artifact in tool-stack handling of escape sequences inside Edit's `new_string` parameter, fixed before commit.

## Decisions Made

1. **displayName collected on step 1 (not a separate step).** Both `sessionName` and `displayName` are upfront-identity concerns; matches design_context. Trade: step 1 is denser, but each input is small and the visual layout uses the existing `form-group` pattern (no CSS changes needed).

2. **Default resolution chain: settings → git config user.name → os.userInfo().username → 'Host'.** Matches design_context spec verbatim. The git lookup is best-effort with 1s timeout, try/catch falls through on any error; non-git workspaces silently degrade.

3. **Workspace-scoped persistence (NOT global).** Different workspaces can have different identities (e.g., a user's "work" repo vs "personal" repo can carry different displayName values). Settings update failures are non-fatal — swallowed via `.then(undefined, () => {})`.

4. **HostIdentity allocated in `handleWizardComplete` (not in `createOrShow`).** The secret materializes only when the user commits to creating a session, NOT every time the wizard panel opens (which would happen on every `versioncon.hostSession` invocation, even if the user closes the wizard without proceeding). Reduces secret-lifetime to the live session.

5. **wireHostEvents stores `activeHostIdentity` at module scope but does NOT modify `hostMemberId`.** Current `extension.ts` uses direct host method calls (e.g., `activeHost.kickMember(memberId, 'Kicked by host')` at line 994) — these bypass the host's wire-frame admin gates and operate via public methods that have no requesterId check. There are NO wire-frame admin commands issued by the local UI in the current codebase. So the existing IIFE admin-permission gates at lines 1282/1687/1768 (`currentMemberId !== hostMemberId`) — which compare `currentMemberId === 'local-user'` against `hostMemberId === 'local-user'` for the host-bypass — still resolve correctly. Pinned by plan 04.1-04 Test 11.
