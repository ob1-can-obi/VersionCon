Test 9 collects all wire envelopes received by Alice (push-notification + chat-message) and asserts the secret is absent. Asserts (a) no server-side throw (length pre-check before `crypto.timingSafeEqual`), (b) `auth-response.accepted === true` (invite-code is the auth gate, not the secret), and (c) the resulting member's role is `'member'`. T-04.1-02-01 mitigation evidence pinned.

Plan 04.1-03 T4 explicitly preserved this decoupling — `SessionHost.this.hostMemberId` is the pre-allocated UUID (wire-side identity), while `extension.ts hostMemberId` is a SEPARATE placeholder used only for the IIFE admin-bypass at lines ~1282/~1687/~1768. If a future change updates the extension.ts placeholder to `hostIdentity.memberId`, the IIFE bypass breaks (`currentMemberId` 'local-user' !== `hostMemberId` UUID → host's own push/branch commands gated). Test 11 catches this regression.

## Decisions Made

1. **Source-grep test for wizard contract (Test 1).** Per STATE.md decisions table — `[Plan 04-11]: UI-SPEC literal verification via source-grep tests`. Webview behavior is verified at the source level; the actual webview-instantiation behavior is deferred to multi-window UAT. Trade: regex-based grep can miss subtle behavioral defects (e.g. a refactor that moves the helper but preserves the literal). Acceptance: source-grep is documented as a complement to multi-window UAT, not a replacement.
2. **Per-test fresh tmpDir + chatLog.** Each test that needs persistence creates its own tmpDir under `os.tmpdir()` with a `Date.now() + Math.random()` suffix, loads a fresh ChatLog, attaches it via `host.setChatLog(chatLog, 'main')`, and `await fs.rm(tmpDir, { recursive: true, force: true })` in teardown. Mirrors 'Phase 4 host relay' setup pattern. Avoids cross-test leakage.
3. **Test 8 + Test 9 secret-hygiene assertions are positive coverage.** Each test broadcasts/observes events and asserts the secret string is NOT in the serialized output. If a future refactor introduces a code path that DOES leak the secret, these tests would catch it at CI time. Sufficient for v1.
4. **Test 10 explicit length-mismatch case.** A 5-char secret (vs. UUID's 36-char) would throw inside `crypto.timingSafeEqual` if the length pre-check were removed. The test sends a raw WebSocket auth-request and asserts (a) no server throw, (b) `accepted: true`, (c) `role: 'member'`. Defense-in-depth evidence for T-04.1-02-01 mitigation.
5. **No webview integration test.** The wizard's webview boundary requires a real `vscode.window.createWebviewPanel` instance and DOM-side message-channel handling — both unavailable in the unit-test runner. Multi-window UAT is the comprehensive layer per Phase 4 convention. Plan 04.1-03's UAT scenarios are documented in that plan's SUMMARY.
