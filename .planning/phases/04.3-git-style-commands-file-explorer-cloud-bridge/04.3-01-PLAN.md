<objective>
Add the UX surface layer that makes VersionCon feel git-like and stops surfacing the internal `.versioncon/` directory in the VS Code File Explorer:

1. On first activation per workspace, merge `{ ".versioncon": true }` into `<workspace>/.vscode/settings.json` under `files.exclude` without clobbering any existing keys. Gate behind a workspaceState flag so we never re-run.
2. Register 7 git-style command aliases (`versioncon.cmd.push`, `versioncon.cmd.pull`, `versioncon.cmd.checkout`, `versioncon.cmd.branch`, `versioncon.cmd.log`, `versioncon.cmd.diff`, `versioncon.cmd.merge`) as thin pass-throughs to the existing canonical commands. This phase does NOT change handler behavior — it adds discoverable Command Palette entries with familiar names.

Purpose: Phase 4 multi-window UAT 2026-05-11 surfaced that the `.versioncon/branches/{branch}/` model is confusing and that users want git-vocabulary handles. This plan delivers SC-1 (files-exclude injection) and SC-2 (git-style aliases) from 04.3-SPEC.md — the smallest, lowest-risk piece of the phase that does not touch the push pipeline.

Output: Hidden `.versioncon/` in File Explorer + 7 new Command Palette entries. No regression to existing 350 tests.

<task type="auto">
  <name>Task 2: Inject files.exclude on first activation via merge-write helper (T-04.3-01 mitigation)</name>
  <files>src/extension.ts</files>
  <action>
1. In `src/extension.ts`, add a top-level helper above `activate()` (near `formatPushToast`):
   ```typescript
   /**
    * Phase 4.3 SC-1: hide .versioncon/ from VS Code's native File Explorer by
    * merging `{ ".versioncon": true }` into <workspace>/.vscode/settings.json
    * under `files.exclude`. T-04.3-01 mitigation: NEVER overwrite the whole
    * file — read existing JSON, deep-merge under files.exclude only, write back.
    * Gated by workspaceState so we never re-run; the user may later remove the
    * key manually and we must respect that.
    */
   async function ensureVersionconExcluded(context: vscode.ExtensionContext): Promise<void> {
     if (context.workspaceState.get<boolean>('versioncon.filesExcludeInjected')) return;
     const folder = vscode.workspace.workspaceFolders?.[0];
     if (!folder) return; // no folder open — defer until one is opened
     const settingsPath = path.join(folder.uri.fsPath, '.vscode', 'settings.json');
     let existing: Record<string, unknown> = {};
     try {
       const raw = await fsPromises.readFile(settingsPath, 'utf-8');
       existing = JSON.parse(raw) as Record<string, unknown>;
     } catch {
       /* file missing or malformed — treat as empty, do NOT propagate */
     }
     const filesExclude = (existing['files.exclude'] as Record<string, unknown> | undefined) ?? {};
     // Do nothing if the key is already set (user may have set false intentionally).
     if (Object.prototype.hasOwnProperty.call(filesExclude, '.versioncon')) {
       await context.workspaceState.update('versioncon.filesExcludeInjected', true);
       return;
     }
     filesExclude['.versioncon'] = true;
     existing['files.exclude'] = filesExclude;
     await fsPromises.mkdir(path.dirname(settingsPath), { recursive: true });
     await fsPromises.writeFile(settingsPath, JSON.stringify(existing, null, 2), 'utf-8');
     await context.workspaceState.update('versioncon.filesExcludeInjected', true);
   }
   ```

In `activate()`, invoke `void ensureVersionconExcluded(context);` early — right after the existing `if (!context.workspaceState.get('versioncon.movedToAux'))` block (around current line ~231) — fire-and-forget IIFE-style so activation never blocks on disk I/O. Wrap any error in `console.error('[versioncon] files.exclude injection failed', err)` so failure is observable but does not break activation. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| VS Code extension → workspace `.vscode/settings.json` | We write to a file the user owns; risk = clobbering unrelated keys |
| User → Command Palette | Aliases are pure pass-throughs; no new attack surface beyond what canonical commands already permit |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04.3-01 | Tampering | `ensureVersionconExcluded` writing to `.vscode/settings.json` | mitigate | Read-merge-write with structured JSON (`JSON.parse` then assign one key under `files.exclude` then `JSON.stringify` — never overwrite the whole file). Gate behind `context.workspaceState.get('versioncon.filesExcludeInjected')` so we run at most once per workspace. If user manually sets the key (true OR false), we do NOT touch it on subsequent activations. Swallow malformed-JSON errors and continue; never propagate to break activation. Defense-in-depth: outer `void ensureVersionconExcluded(...).catch(console.error)` so write failures never break extension activation. |
| T-04.3-02 | Information disclosure | Alias commands could leak intent of canonical commands to unauthorized callers | accept | Aliases route via `vscode.commands.executeCommand` — VS Code applies the same permission gates the canonical commands enforce (canPushToBranch, canCreateBranch, etc.). No new attack surface. |
| T-04.3-05 | Repudiation | Alias logs nothing distinct from canonical command, making it hard to attribute "user ran alias vs canonical" | accept | Low value — both eventually call the same handler which already logs via push history / activity log. Not worth telemetry plumbing for v1. |
</threat_model>
