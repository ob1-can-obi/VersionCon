# Phase 4.3 Plan 04: Cloud Bridge — Export to / Import from Git Remote — Summary

Cloud-bridge surface (SPEC SC-6 + SC-7) — the LAN → cloud handoff that closes the lifecycle loop. A host with admin permission can ship the active VersionCon branch to a real git remote (GitHub, GitLab, local bare repo) or pull a remote down as a fresh VersionCon branch. All `git` invocations go through a shell-safe `GitBridge` service that uses `child_process.spawn` with `shell: false` and an argv array — URL + branch-name inputs are validated against allowlist regexes before they ever reach spawn (T-04.3-03 command-injection mitigation). 

## What shipped

1. **GitBridge service** (`src/services/GitBridge.ts`, 241 lines). Exports `validateUrl`, `validateBranchName`, `runGit`, `exportToRemote`, `importFromRemote`. All git invocations through `child_process.spawn('git', argv, { cwd, shell: false })` — **never** string concatenation, **never** `exec()` / `execSync()`. Input validation via allowlist regexes for https://, git@host:, file:// URLs and `[A-Za-z0-9._/-]{1,128}` branch names.

2. **versioncon.exportToGitRemote** (host-only, admin-gated, SC-6). Runs `git init` + `git add .` + `git commit -m <msg>` + `git remote add origin <url>` + `git push -u origin <branch>` inside `.versioncon/branches/{branch}/`. Two-layer gate: `!activeHost` (host-only) AND `!permissions.canCreateBranch(currentMemberId)` (admin proxy with host-bypass). T-04.3-02 confirmation modal before push fires, showing URL + branch + file count + commit message.

3. **versioncon.importFromGitRemote** (host-only, admin-gated, SC-7). Runs `git clone --branch <b> --single-branch <url> <dest>` into a fresh `.versioncon/branches/{newName}/`, strips `.git/` post-clone (T-04.3-13 — prevents malicious hooks), and registers via `BranchManager.registerExternalBranch`. Same gates as export. Confirmation modal mirrors export for symmetry.

4. **BranchManager.registerExternalBranch** — registers an already-populated branch dir without copying contents. Relaxed validator accepts dots + slashes (`feature/v1.0` etc.) so git-style names round-trip. Rejects missing-on-disk + name collisions. Persists via `saveMetadata`.

5. **Output channel `VersionCon: Git Bridge`** — lazily constructed singleton; surfaces streaming `git` stdout/stderr to the user. Pushed to `context.subscriptions` once so it disposes with the extension.

6. **45 new tests** in `src/test/suite/gitBridge.test.ts`:
   - 13 validator allow/reject (URL + branch name)
   - 2 runGit smoke (`--version` success, nonexistent subcommand failure)
   - 3 GitBridge round-trip (export, import, export→import combined)
   - 3 spawn-invariant source-grep (T-04.3-03: shell:false explicit; no exec/execSync; module structure)
   - 10 exportToGitRemote wiring source-grep (import, registration, both gate layers, validators, modal-before-export, channel name, catch path, package.json)
   - 9 importFromGitRemote wiring source-grep (registration, both gates, collision check, modal, registerExternalBranch call, cleanup-on-error, package.json, BranchManager method declared)
   - 5 BranchManager.registerExternalBranch behavior (missing dir, collision, bad name, success + persistence, dots/slashes accepted)

## Decisions made

1. **URL allowlist scoped to https/ssh-shorthand/file** — plan §allowlist_regex explicitly defers ssh:// to v2. The three accepted schemes cover real-world GitHub/GitLab (https) + power users (git@) + tests (file://). Anything else (javascript:, data:, custom URI schemes) is rejected before spawn.

2. **Two validators for branch names by design** — `BranchManager.createBranch` keeps the stricter `^[a-zA-Z0-9_-]+$` rule for VersionCon-internal branches; `BranchManager.registerExternalBranch` allows dots and slashes (`feature/v1.0`) to match what real git remotes look like. JSDoc on `registerExternalBranch` explicitly cross-references the contrast.

3. **`git commit` "nothing to commit" soft-success** — exit 1 with that phrase in stdout OR stderr is treated as success and the push proceeds against HEAD. Re-running export after a no-op edit therefore still updates origin without forcing the user to fabricate a change. Plan §deviation_protocol explicitly authorizes this relaxation; I extended it to also detect `Please tell me who you are` and surface a helpful instruction (Rule 2 hardening).

4. **Source-grep block-extractor terminator** — initial regex `\}\)/` was non-greedy and stopped at the first `})` (the close of the `validateInput` arrow function inside `showInputBox`), missing `validateBranchName` / modal / catch downstream. Switched to `\}\),\s*\n\s*\);` — the close of `context.subscriptions.push(registerCommand(...))`. Reusable for any future source-grep test that needs to inspect a whole registerCommand block.

5. **Output channel lazily constructed** — created on first invocation rather than at activate-time. Two reasons: (a) keeps the Output dropdown clean for users who never touch the cloud bridge; (b) avoids one more `context.subscriptions.push` at activate that runs even when no workspace is open. The `gitBridgeChannelPushedToSubs` flag guards against duplicate-subscription on repeat invocations.

6. **Concurrent Wave 2 edit captured Wave 4 lines (Rule 1 deviation, documented below)** — commit `3b79f6b` (Wave 2's WorkspaceDiffer-into-push wiring) accidentally staged + committed two of my uncommitted Wave 4 edits to `src/extension.ts`. Behavior is correct; commit attribution is wrong but not destructive.

## Deviations from plan

### Auto-fixed Issues

**1. [Rule 2 — defensive hardening] git commit "Please tell me who you are" error → helpful instruction**
- **Found during:** Task 1 round-trip test design.
- **Issue:** Plan §deviation_protocol explicitly anticipated this case ("relax the gate to also accept exit 1 when stderr matches `/(nothing to commit|Please tell me who you are)/` and surface the helpful instruction"). The default git error is opaque to non-git users.
- **Fix:** exportToRemote catches the matched stderr/stdout and throws a new Error with the exact `git config --global user.email ... user.name ...` recipe. Tests cover both the soft "nothing to commit" path and normal failure modes; the user-config branch is hard to trigger inside the test harness (we set local user.* in the round-trip setup), but the error message will surface verbatim when it does fire.
- **Files modified:** `src/services/GitBridge.ts` (exportToRemote commit-error branch).
- **Commit:** `94d6f44`.

**2. [Rule 1 — bug fix in test regex] Block extractor terminator widened**
- **Found during:** Task 2 test run (3 of 10 wiring tests failed: validateBranchName, modal, catch).
- **Issue:** My initial regex `/registerCommand\(\s*['"]versioncon\.exportToGitRemote['"][\s\S]*?\n\s{0,12}\}\)/` was non-greedy with a generic `\}\)` terminator, stopping at the first close-paren-after-brace it encountered — which was the close of `validateInput: (v) => ... })`, not the close of the whole registerCommand block. As a result, the captured block fragment missed everything below the URL prompt.
- **Fix:** Switched terminator to `\}\),\s*\n\s*\);` — matches the literal `}),\n      );` close of `context.subscriptions.push(registerCommand(...))`. After the swap, all 10 wiring tests passed.
- **Files modified:** `src/test/suite/gitBridge.test.ts` (4 source-grep tests in the exportToGitRemote suite).
- **Commit:** `3bd57cc` (the fix shipped together with the production code in Task 2's commit; tests went RED before the production code landed and GREEN after).

**3. [Rule 0 — concurrent execution] Wave 2 commit absorbed Wave 4 lines**
- **Found during:** Task 2 staging (post-Task-1, post-Wave-2-Task-2-commit `3b79f6b`).
- **Issue:** While I was writing Task 1's GitBridge tests, Wave 2 was concurrently working on its Task 2 (WorkspaceDiffer-into-push wiring). Wave 2's commit `3b79f6b` apparently used a greedy `git add src/extension.ts` (or equivalent) and captured TWO of my uncommitted Wave 4 edits in addition to its own: `import { GitBridge } from './services/GitBridge.js';` and the entire `gitBridgeOutputChannel` + `getGitBridgeOutputChannel` block. The commit message of `3b79f6b` only mentions the WorkspaceDiffer wiring.
- **Disposition:** **Leave it alone.** Rewriting history with `git commit --amend` or `git rebase -i` would be destructive (Wave 2 may already be downstream of `3b79f6b`). The lines are functionally correct AND in HEAD. Wave 4 Task 2's commit `3bd57cc` lands the actual `registerCommand('versioncon.exportToGitRemote', ...)` block + package.json entry on top. The mis-attribution is cosmetic.
- **Mirror:** Wave 2's commit `9c57fd9` documented the same scenario in reverse — "concurrent Wave 4 export-to-git commit 3bd57cc (which staged extension.ts greedily). Functionality is correct and lives in HEAD." Both waves saw the same race.
- **Lesson for future parallel waves:** When two executors share a single file (here `src/extension.ts`), `git add src/extension.ts` is unsafe because the working tree may contain uncommitted edits from the other executor. Use `git add -p` (interactive patch staging) or commit immediately after each edit. The plan's execution_mode warning was correct in principle but neither executor adopted patch-staging.
- **Files affected:** `src/extension.ts` (3 lines absorbed into `3b79f6b` that belonged in Wave 4's Task 2 commit `3bd57cc`).
