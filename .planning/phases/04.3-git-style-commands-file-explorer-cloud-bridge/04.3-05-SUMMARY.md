## Decisions made

1. **README.md created from scratch (Rule 2 deviation).** The plan's deviation_protocol explicitly anticipated this: "If the README does not yet exist (currently a stub), Rule 2: create it with the minimum structure described in Task 1, plus a one-line elevator pitch lifted from PROJECT.md's 'Core Value' section so the new sections have context." `ls README*` confirmed no README at any case-variation. The new file uses the minimum structure the plan specified: title, elevator pitch (verbatim from PROJECT.md What This Is + Core Value), Status, Install (development), Lifecycle Tour, VersionCon for Git Users, Development. No screenshots or GIFs added (out of scope per plan §DO NOT).

2. **VersionCon-for-Git-Users table columns merged.** The plan template had a 5-column shape (Git | VersionCon command (Command Palette) | Git-style alias | Canonical command id | Notes). Looking at the shipped package.json, the cmd.* aliases ARE the user-facing `VersionCon: Push` / `VersionCon: Pull` / etc. titles, while the canonical `versioncon.push` is titled `Push to Branch` (no VersionCon: prefix — it's the context-menu surface from Phase 3, not a Command Palette entry under VersionCon:). So the plan's two columns ("VersionCon command" + "Git-style alias") would have shown the same string. I merged them into a single column showing the user-visible title once, and put both ids (`versioncon.cmd.push → versioncon.push`) in the canonical-id column for scripting clarity. Per deviation_protocol: "the table is the contract, the code is the source of truth."

3. **PROJECT.md mirror is terser AND cross-references README.** End-users want the Alice/Bob narrative; phase planners want the phase-anchored functional summary. Splitting the audiences keeps each doc concise. The PROJECT.md Lifecycle Tour ends with "See `README.md` for the full user-facing walkthrough" so a planner who wants the long form has the pointer.

4. **PROJECT.md edit is NOT committed.** Constraints in the prompt and the gitignore both say `.planning/` is gitignored; the plan does not direct `git add -f` for this file. The edit lives on disk for the orchestrator (which is expected to handle gitignored planning artifacts via the GSD SDK at the phase-completion step). The Task 2 row in the table above documents this explicitly.

5. **Section placement matches plan literally.** Lifecycle Tour goes AFTER `## Constraints` and BEFORE `## Key Decisions`. Order of top-level sections is now: What This Is → Core Value → Requirements → Context → Constraints → **Lifecycle Tour** → **VersionCon-for-Git-Users vocabulary** → Key Decisions → Evolution. (`## Out of Scope` is rendered inside `## Requirements` as a subsection in the original file; left untouched per plan invariant.)
