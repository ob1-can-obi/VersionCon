# Phase 4.3 Spec: Git-Style Commands + File Explorer Workflow + Cloud Bridge

## Background

Phase 4 multi-window UAT surfaced a recurring UX pain point: the `.versioncon/branches/{branch}/` model is unfamiliar, the split-pane drag-and-drop feels mandatory, and there's no story for "we finished, now ship the project to GitHub." User feedback (verbatim):

> "Alice and Bob start off by creating a folder named PROJECT and start working on it ... Alice makes a branch in LAN. Bob takes it and both make changes and work normally as usual. Bob pushes his changes and Alice hers ... the pushing and all let them be through command prompt so it will be easier. Let it also mimic git so people understand better. Finally Alice who has the permissions, will take the final branch after testing - will push it to the cloud or wherever they want to use the code at."
>
> "I also want them to be able to use the File Explorer directly if they want to but changes must be local and can only be changes through LAN commits or pushes."

This phase wraps a **UX layer** around the existing collaboration engine — no architectural pivot. PROJECT.md's "branch as source of truth, workspace as scratch pad" decision is **preserved**.

## What this phase DOES change

1. **Hide `.versioncon/` from File Explorer.** Auto-inject `files.exclude: { ".versioncon": true }` into `<workspace>/.vscode/settings.json` on first activation. The directory is implementation detail — users see only their project files in the native File Explorer.

2. **Git-style command aliases.** Add `versioncon.push`, `versioncon.pull`, `versioncon.checkout`, `versioncon.branch`, `versioncon.log`, `versioncon.diff`, `versioncon.merge` as Command Palette entries. Each routes to the existing handler under a familiar name. Categories all `VersionCon: ...` so Command Palette grouping is clean.

3. **Workspace-diff–driven push/pull (no drag-required).** `versioncon.push` diffs the workspace against `.versioncon/branches/{branch}/` (excluding `.versioncon/`, `.vscode/`, common ignore patterns) and pushes ALL changed files through the existing push pipeline. `versioncon.pull` copies everything from `.versioncon/branches/{branch}/` into the workspace, overwriting (with the existing per-file conflict prompt from Plan 03-06).

4. **Status-bar "uncommitted changes" indicator.** When workspace has changes relative to active branch, status bar reads e.g. `$(git-pull-request) 3 local changes`. Click → `versioncon.diff` shows a QuickPick preview of what would be pushed. Hidden when zero changes.

5. **`versioncon.exportToGitRemote`** (host-only, admin-gated). Prompts host for: remote URL (string), commit message (string), branch name on remote (default = current VersionCon branch name). Runs `git init` / `git add .` / `git commit -m "..."` / `git remote add origin <url>` / `git push -u origin <branch>` inside `.versioncon/branches/{branch}/`. Surfaces command output in an Output channel `VersionCon: Git Bridge`.

6. **`versioncon.importFromGitRemote`** (host-only, admin-gated). Prompts host for: remote URL, branch name to clone. Runs `git clone <url> -b <branch> <tmp>` and copies content into a fresh `.versioncon/branches/{newBranchName}/`. Registers the new branch via BranchManager. Use case: starting v2 of a project after an initial export.

7. **Docs.** Update README and PROJECT.md with a "Lifecycle Tour" that walks through create → collab → export → next-version. Add a "VersionCon for Git users" quick-ref table mapping VersionCon verbs to git equivalents.

## What this phase does NOT change

- Split-pane drag-and-drop UI — stays as a power-user surface.
- `.versioncon/branches/{branch}/` storage model — unchanged. Branch is still source of truth.
- The configurable `.versioncon/` location (deferred from 999.5) — still hard-locked to `<workspace>/.versioncon/`.
- Auto-pushing to cloud git on every LAN push — cloud export is a deliberate host-only action.
- The host-vs-member permission model — `exportToGitRemote` and `importFromGitRemote` use existing admin permission gate.

## Success Criteria (testable)

1. **Files-exclude injection**: open a fresh workspace, activate VersionCon → `<workspace>/.vscode/settings.json` contains `"files.exclude": { ".versioncon": true }` merged with any existing settings (no overwrite of unrelated keys). VS Code File Explorer does not show `.versioncon/`.

2. **Git-style command aliases**: Command Palette → type `VersionCon: Push` → executes the same push pipeline that the existing `versioncon.commit` (or equivalent) handler runs. Same for pull, checkout, branch, log, diff, merge. Source-grep test asserts all 7 commands are declared in `package.json` `contributes.commands`.

3. **Workspace-diff push**: with `vc push`, edit a file in the workspace (no drag-and-drop), run the command → the changed file is copied to `.versioncon/branches/main/`, push-history entry created, broadcast to peers visible in their activity log within 2s.

4. **Workspace-diff pull**: another member pushes a file edit. Run `vc pull` → the workspace file is overwritten with the branch version. If there's a local conflict (workspace edit on the same file), the existing per-file conflict prompt (PUSH-11) appears.

5. **Status-bar indicator**: edit a file in the workspace → status bar shows `$(git-pull-request) 1 local change` within 500ms of save. Click it → QuickPick lists the 1 changed file. After `vc push`, the indicator hides.

6. **Cloud export**: as host, run `versioncon.exportToGitRemote` against a local bare repo URL (e.g. `file:///tmp/test-bare.git`) → `git log` in that bare repo shows the commit; the branch is pushed.

7. **Cloud import**: as host, run `versioncon.importFromGitRemote` with a URL pointing at the bare repo just created + a new branch name → BranchManager lists the new branch; its `.versioncon/branches/{name}/` contains the files.

8. **No regression**: all 350 existing tests pass; drag-and-drop push/pull still works; chat/presence/notifications unaffected.

## Wave breakdown (planner consumes this)

- **Wave 1**: `files.exclude` injection + git-style command aliases. Small, mostly `package.json` + a thin `commands.ts` re-export.
- **Wave 2**: Workspace-diff push/pull. Non-trivial: needs a `WorkspaceDiffer` service (compute changed files from workspace vs. branch dir), plus integration into the existing push pipeline. New tests for the diff logic.
- **Wave 3**: Status-bar indicator + `versioncon.diff`. Subscribes to FileSystemWatcher debounce events, runs the differ, updates a `StatusBarItem`.
- **Wave 4**: Cloud bridge — `versioncon.exportToGitRemote` + `versioncon.importFromGitRemote`. Uses `child_process.spawn('git', [...])`. Output channel for visibility. Admin-permission gate.
- **Wave 5**: Docs + UAT. README "Lifecycle Tour" + "VersionCon for Git users" quick-ref. UAT plan.

## Threat model notes (for planner to expand)

- **T-04.3-01**: Auto-injecting `files.exclude` into a user's `.vscode/settings.json` is a workspace-level write. Risk: clobbering existing keys. Mitigation: read-merge-write with structured JSON; never overwrite the whole file; gate behind a one-shot workspaceState flag so we don't re-run on every activation.
- **T-04.3-02**: `git push` to a user-supplied URL — could be abused to exfil. Mitigation: clear confirmation dialog before push, show the URL + branch name + commit count in the prompt, log to Output channel.
- **T-04.3-03**: `child_process.spawn('git', ...)` with user input — command injection risk. Mitigation: spawn with argv array (NEVER shell:true), validate URL/branch-name args against allowlist regex.
- **T-04.3-04**: `versioncon.exportToGitRemote` permission. Must check `permissions.isAdmin(currentMemberId)` before running. Source-grep test pins the check.
