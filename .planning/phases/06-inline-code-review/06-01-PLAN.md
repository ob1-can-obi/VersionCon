<objective>
Wave 1 of Phase 6: lay down every type, wire-protocol discriminant, branch-metadata extension, and persistence module the next four waves consume — in one pass, before any host/client/UI code lands. This is the contract-first wave Phase 5 used for its `src/ast/types.ts` + `AstFactory.ts` foundation (see 05-01-PLAN.md); we mirror that posture so Waves 2-5 can build in parallel paths against stable shapes.

Concretely, this plan delivers:

1. **`src/types/review.ts`** — the canonical Phase 6 type module: `ReviewVote` ('approved' | 'changes-requested' | 'commented'), `ReviewStatus` ('open' | 'approved' | 'changes-requested' | 'resolved' | 'abandoned'), `ReviewComment`, `ReviewVoteRecord`, `ReviewRequest` — verbatim from 06-SPEC.md "Data shapes".
2. **`src/types/chat.ts` extension** — extend `SystemEventSubKind` union with 5 new members: `'review-opened' | 'review-comment' | 'review-approved' | 'review-changes-requested' | 'review-resolved'`. Wave 2 (`SessionHost.appendAndBroadcastSystemEvent`) emits chat-message wire frames carrying these sub-kinds; Plan 04-10's `ChatPanel.renderSystemRow` already keys off `record.kind === 'system'` so rendering is gracefully handled — Wave 4 adds Phase 6-specific copy via a sub-kind switch but unknown sub-kinds still fall through to the existing generic system-row renderer.
3. **`src/network/protocol.ts` extension** — extend `ProtocolMessage` union with 5 new discriminated wire types: `review-opened`, `review-comment`, `review-vote`, `review-resolved`, `review-state-sync`. Add each member of the set to `VALID_TYPES` so `parseMessage` admits them (omission = silent drop, which would invisibly regress Wave 3 routing).
4. **`src/types/branch.ts` extension** — add `requireReview?: boolean` to `BranchInfo`. Optional so existing `.versioncon/branch-metadata.json` files load unchanged (REVIEW-04 admin gate read by Wave 5 `mergeBranch` and written by Wave 5 `versioncon.setBranchRequireReview`).
5. **`src/filesystem/ReviewStore.ts`** — per-review JSON persistence at `.versioncon/branches/{branch}/reviews/{pushId}.json`. One file per `ReviewRequest`, indexed in-memory by `pushId`. Mirrors the load-on-activate + write-on-mutation pattern used by `PushHistory` and `ChatLog`. Public API: `load(branchesDir)`, `upsertRequest(req)`, `getReview(pushId)`, `getAll()`, `getOpenForBranch(branch)`.
6. **Tests** — `reviewStore.test.ts` (CRUD round-trip on a real temp dir) and `reviewTypesContract.test.ts` (source-grep that pins the type-shape contract so Waves 2-5 cannot drift).

Why Wave 1 owns all of this: Waves 2-5 are largely parallel work that touches separate subsystems (host, client, webview, extension). Locking the shared types here means a Wave 4 ReviewPanel rebuild against a shifting `ReviewRequest` shape never blocks Wave 2 host relay. The same architectural choice that Phase 5 Wave 1 made and that paid off for Phase 5 — same convention here.

/**

 * prior request: Wave 2 host handler closes the prior review with
 * status:'abandoned' and opens a fresh one for the new pushId (locked
 * decision from 06-SPEC.md frontmatter).

  | 'changes-requested'
  | 'resolved'
  | 'abandoned';

  id: string;                    // uuid (crypto.randomUUID)

}

/**
 * Per-push review unit. Authored by the pusher (authorMemberId === push
 * author at open time); votes + comments stream in over the wire and are
 * persisted to `.versioncon/branches/<branch>/reviews/<pushId>.json` on
 * every mutation (whole-file rewrite, mirrors PushHistory).
 */

  id: string;                    // uuid

  openedAt: number;
  status: ReviewStatus;

  comments: ReviewComment[];
  resolvedAt?: number;
  resolvedBy?: string;           // memberId who resolved (push author OR admin override)
  resolvedReason?: 'merged' | 'abandoned';
}

import * as fs from 'fs/promises';
import * as path from 'path';
import type { ReviewRequest } from '../types/review.js';

/**
 * Per-review JSON persistence at .versioncon/branches/{branch}/reviews/{pushId}.json.
 *
 * Storage layout:
 *   .versioncon/
 *     branches/
 *       main/
 *         reviews/
 *           abc123.json   ← one ReviewRequest with pushId 'abc123'
 *           def456.json   ← one ReviewRequest with pushId 'def456'
 *
 * Persistence pattern mirrors PushHistory + ChatLog (Phase 3 + 4):
 *   - load on activate (per-branch scope — call load(branchName) per branch)
 *   - write WHOLE file on every mutation (no incremental diff; same
 *     posture as ChatLog.append which rewrites the whole chat-log.json)
 *   - in-memory index by pushId for O(1) lookup
 *
 * The store is branch-scoped: load(branchName) populates the in-memory
 * index for ONE branch. Wave 5 mergeBranch needs reviews from the SOURCE
 * branch to gate the merge into the TARGET — extension.ts orchestrates by
 * calling load(sourceBranch) before the gate check.
 *
 * Threat T-06-04 (file-system bypass): out of scope — same posture as
 * Phase 3 SyncTracker. A malicious user with OS-level write access to
 * .versioncon/branches/{branch}/reviews/*.json can craft fake approved
 * reviews. VersionCon's threat model assumes the disk is trusted; we are
 * not a defense against attackers with shell access to your machine.
 */
export class ReviewStore {
  /** memoized by pushId (the natural per-review key per 06-SPEC.md "Storage"). */
  private readonly reviewsByPushId: Map<string, ReviewRequest> = new Map();

  constructor(private readonly versionconDir: string) {}

  /**
   * Load every ReviewRequest under .versioncon/branches/{branchName}/reviews/
   * into the in-memory index. Missing dir → no-op (returns 0). Corrupt file
   * → console.error + skip (does not abort the load). Call on activate AND
   * on every branch switch.
   *
   * Returns the count of reviews loaded for the requested branch.
   */
  async load(branchName: string): Promise<number> {
    const dir = path.join(this.versionconDir, 'branches', branchName, 'reviews');
    let entries: string[];
    try {
      entries = await fs.readdir(dir);
    } catch {
      return 0;
    }
    let loaded = 0;
    for (const entry of entries) {
      if (!entry.endsWith('.json')) continue;
      const full = path.join(dir, entry);
      try {
        const raw = await fs.readFile(full, 'utf-8');
        const req = JSON.parse(raw) as ReviewRequest;
        if (typeof req.pushId === 'string' && typeof req.id === 'string') {
          this.reviewsByPushId.set(req.pushId, req);
          loaded += 1;
        }
      } catch (err) {
        console.error(`[ReviewStore] failed to load ${full}`, err);
      }
    }
    return loaded;
  }

  /**
   * Persist a ReviewRequest. Writes the WHOLE file (no diff). Updates the
   * in-memory index. Last write wins (mutation flows through this method
   * for both first-write and updates).
   */
  async upsertRequest(req: ReviewRequest): Promise<void> {
    const dir = path.join(this.versionconDir, 'branches', req.branch, 'reviews');
    await fs.mkdir(dir, { recursive: true });
    const file = path.join(dir, `${req.pushId}.json`);
    await fs.writeFile(file, JSON.stringify(req, null, 2));
    this.reviewsByPushId.set(req.pushId, req);
  }

  /**
   * Defensive deep copy via JSON round-trip — mutating the returned object
   * cannot corrupt the in-memory index. Mirrors PresenceMap.getSnapshot's
   * defensive-copy invariant (STATE.md decision from Plan 04-03).
   */
  getReview(pushId: string): ReviewRequest | undefined {
    const r = this.reviewsByPushId.get(pushId);
    return r ? JSON.parse(JSON.stringify(r)) as ReviewRequest : undefined;
  }

  /** All known reviews as a defensive copy. Ordering not guaranteed. */
  getAll(): ReviewRequest[] {
    return Array.from(this.reviewsByPushId.values()).map(
      r => JSON.parse(JSON.stringify(r)) as ReviewRequest,
    );
  }

  /** Open reviews for the given branch. Filters by status === 'open'. */
  getOpenForBranch(branch: string): ReviewRequest[] {
    const out: ReviewRequest[] = [];
    for (const r of this.reviewsByPushId.values()) {
      if (r.branch === branch && r.status === 'open') {
        out.push(JSON.parse(JSON.stringify(r)) as ReviewRequest);
      }
    }
    return out;
  }
}
